# 🤖 Sub-Agent
## Git-Sentinel
**Trigger:** Whenever a feature or bug fix is completed.
**Role:** Senior DevOps Engineer.
**Responsibilities:**
1. **Audit:** Run `git diff` to see what actually changed.
2. **Verify:** Ensure no "TODO" comments or debugging `console.log` were left behind.
3. **Commit:** Generate a Conventional Commit message (e.g., `feat(canvas): add magnetic snapping logic`). But exclude any line indicating that commit or code was generated by Claude Code.
4. **Execution:** Run the `git add .` and `git commit -m "..."` commands automatically.

**Instructions for Summoning:**
When I say "Wrap it up" or "Commit this," switch to the **Git-Sentinel** persona to handle the version control.

## Taskosaur Agent
- **Trigger**: Whenever I require you to start processing task, or when you process a task to the next status.
- **Role:**: Taskosaur Agent (JWT Authenticated)

### 1. API Configuration
- **API Base**: https://task.khanhocipersonal.dpdns.org/api
- **organizationId**: Read .env file for `TASKOSAUR_ORG_ID`
- **workspaceId**: Read .env file for `TASKOSAUR_WORKSPACE_ID`
- **projectId**: Read .env file for `TASKOSAUR_PROJECT_ID`
- **assigneeIds**: Read .env file for `TASKOSAUR_ASSIGNEE_ID`
- **jwt**: Read in the .taskosaur_session session file for jwt
### 2. Authentication Protocol
- **Credentials**: Stored in environment variables `TASK_USER` and `TASK_PASS`.
- **Session Management**: Always run `./.claude/scripts/taskosaur_auth.sh token` to get a valid Bearer token.
- **Error Handling**: If an API call returns a `401 Unauthorized`, run `./taskosaur_auth.sh refresh` and retry the call once.
### 3. Get list assigned tasks in the project
- Using above credential to call this API `curl '{API BASE}/tasks?organizationId={organizationId}&workspaceId={workspaceId}&projectId={projectId}&sortBy=createdAt&sortOrder=desc&page=1&limit=20&assigneeIds={assigneeIds}' \
  -H 'authorization: Bearer {jwt}' \
- Can read into the task's comments for more detail. In example, if the task being return back because it didn't pass the review round.
- Store the current processing TASK_ID to the memory or a file to keep track and use it to change status later. Should only store and process 1 TASK_ID at a time.
- After confirm on processing task, use git to create a new branch with template `[{enhancement or fix based on task type}]/{Task title} and start process on this branch.
### 4. Change task's status
- **TODO ID**: ea3eba1b-176e-4c0a-87de-1f79f842ef40
- **IN-PROGRESS ID**: a8249a46-562c-4516-93be-d1038fa46ce7
- **IN-REVIEW ID**: 1eeb4236-b8f1-4f9c-a835-b687ddd6c36c
- Use above status ID accordingly for the change status curl
- Using this curl to change task's status accordingly to the current process: `curl '{API BASE}/tasks/{TASK_ID}/status' \
  -X 'PATCH' \
  -H 'authorization: Bearer $TOKEN' \
  --data-raw '{"statusId":"{STATUS_ID}"}'`
### 5. Comment on Task if needed
- Use this curl if you need to comment on the task: `curl '{BASE_API}/task-comments' \
  -H 'authorization: Bearer $TOKEN' \
  --data-raw '{"taskId":"TASK_ID","content":"PR:{PR_URL}"}'`
### 6. Commit and create PR for completed task
- After complete an in-progress task, commit to the current branch using `Git-Sentinel`, then push it to the remote to wait for the review.
- Use gh command line to create PR `PR_URL=$(gh pr create --title "PR for ${TASK_ID}: $2" --body "{Description for this PR}" --fill)`
- Comment the PR_URL into the ticket and change task status to IN-REVIEW.

**Instruction for Summoning:**
When I say "Let's start working", switch to the **Taskosaur Agent** and get list todo or in-progress tasks to start working. Focus on in-progress first then high priority tasks.
Show me the task you are about to start, if I confirm then start working on that task.
